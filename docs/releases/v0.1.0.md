# otel-policy-lab v0.1.0

**Catch dangerous OpenTelemetry Collector config changes in CI — before they drop error traces, leak secrets, or blow up your metrics bill.**

First public release of `otel-policy-lab`: a fast, fixture-based guardrail that runs representative OTLP telemetry through your Collector config and checks the output against a separate policy file.

## Highlights

- **Policy-as-code for telemetry governance** — keep Collector config (`how`) and policy (`what must hold`) in separate, reviewable files
- **CI-friendly pass/fail output** — terminal summary plus JSON report with non-zero exit on policy failure
- **GitHub Action** — composite action that builds and runs checks in your workflow
- **Honest confidence model** — warnings when the fixture runner only partially simulates processors or cannot prove span-dropping behavior
- **Safe-by-design reports** — forbidden-value matches never store raw secrets in JSON output

## What's included

### Policy checks

- Forbidden attribute keys (logs, traces, metrics) with optional case-insensitive matching
- Forbidden value patterns (log bodies, span attributes/names, metric label values)
- `forbidden_label_value_patterns` for metric label value scanning
- Required resource attributes (e.g. `service.name`, `deployment.environment`)
- Error trace preservation checks (`status.code == "ERROR"`)
- Metric cardinality limits (`max_series_per_metric`)

### Fixture runner (MVP)

Simulates a documented subset of Collector processor behavior:

- `attributes/*` delete actions on resource attributes, signal attributes, and metric labels
- Narrow debug log filtering
- `batch` treated as a no-op

Emits warnings for unsupported or partially simulated processors instead of silently skipping them.

### GitHub Action

```yaml
steps:
  - uses: actions/checkout@v4
  - uses: andyphied/otel-policy-lab@v0.1.0
    with:
      collector-config: ./collector.yaml
      input: ./fixtures/checkout.otlp.json
      policy: ./policy.yaml
      report: ./otel-policy-report.json
```

Optional inputs: `fail-on-warn`, `validate-collector`, `otelcol-bin`.

### CLI

```sh
otel-policy-lab test \
  --collector-config ./examples/collector.yaml \
  --input ./examples/fixtures/checkout.otlp.json \
  --policy ./examples/policy.yaml \
  --report ./report.json
```

Also supports `otel-policy-lab validate` (delegates to `otelcol validate` when available).

## Security

- JSON reports redact raw forbidden attribute values, log bodies, and full resource attribute maps on failures
- Match details include metadata only: `kind`, `key`, `pattern`, `value_length`, `index`, `metric`

## Known limitations

The default `fixture` runner does **not** execute a real OpenTelemetry Collector binary. It is reliable for output-presence checks (secrets, required attributes, cardinality) and deliberately honest about gaps:

- Over-aggressive log filtering → **warn**, not fail
- Span dropping / sampling → not simulated; trace preservation checks are fixture-local

See [`docs/failure-modes.md`](../../failure-modes.md) and [`docs/design-decisions.md`](../../design-decisions.md).

## Requirements

- Go 1.22.4 (matches `go.mod`)
- OTLP JSON fixtures
- Collector config YAML and separate policy YAML
